# fix022 — Container Bash tool blocked: EROFS read-only filesystem in .claude mount

**Date:** 2026-05-15

## Symptom

Every Bash tool call inside a container session failed with:

```
Error: EROFS: read-only file system, mkdir '/home/dev/.claude/session-env/baf4f44b-...'
```

No shell commands could execute at all. Claude completed code edits but could not
run pytest, ruff, or any other command to verify the changes.

## Root cause

Claude Code v2.1.141+ writes session state (tool outputs, environment snapshots) to
`~/.claude/session-env/<session-id>/` at startup. The container mounts the host
`.claude` directory read-only:

```python
cmd += ["-v", f"{_to_posix(claude_dir)}:/home/dev/.claude:ro"]
```

The `:ro` flag blocks the write, causing every Bash invocation to fail before it
can execute.

This flag was added intentionally to prevent the container from modifying host
Claude settings and credentials. The assumption was that Claude Code only reads
from `~/.claude` — that was true in older versions but broke in v2.1.141+.

## Fix

Keep `~/.claude` read-only but add an anonymous Docker volume that shadows only
the `session-env` subdirectory, making it writable without exposing the rest:

```python
cmd += ["-v", f"{_to_posix(claude_dir)}:/home/dev/.claude:ro"]
cmd += ["-v", "/home/dev/.claude/session-env"]   # anonymous vol — writable
```

Docker processes mounts in declaration order. The anonymous volume at the more
specific path takes precedence over the `:ro` parent mount for that subdirectory.

## Files changed

- `scripts/learnx_dk.py` — `build_docker_command()`: added anonymous volume line

## Rules for future work

1. After upgrading Claude Code, test a container session by running a simple
   Bash command (`python --version`). If it fails with EROFS, a new version
   has started writing to a new path inside `~/.claude`.
2. Never mount `~/.claude` read-write. The directory contains the API key and
   settings. Use targeted anonymous volumes for any subdirectory that needs writes.
