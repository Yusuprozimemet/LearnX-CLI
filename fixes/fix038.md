# fix038 — Notifier never fires: not wired into execution paths; `enabled()` returns True for incomplete Telegram config

**Date:** 2026-05-16

## Symptom

Configuring `[notify]` in `devloop.toml` had no effect. Notifications were never sent,
regardless of which channels were configured. Additionally, setting only
`telegram_token_env` (without `telegram_chat_id_env`) caused `enabled()` to return
`True`, but `_send_telegram()` would silently skip because the chat ID env var was
absent.

## Root cause

Two independent bugs:

**Bug 1 — notification paths never called:**
`_build_notify_payload()` and `Notifier` were defined in `learnx_dk.py` but no code
path called them after a run completed or failed. Both `run_yolo_version()` and
`main()` returned without constructing a `Notifier` or calling `send()`.

**Bug 2 — `enabled()` checked only token, not chat ID:**
```python
# before
def enabled(self) -> bool:
    return bool(self._webhook_url or self._tg_token_env or self._script)
```
`self._tg_token_env` is truthy when only the token env var name is set. If
`telegram_chat_id_env` is absent, `_send_telegram()` logs "not set" and returns — but
`enabled()` already promised a channel was ready, so `send()` was called pointlessly.

## Fix

**Bug 1 — wire notifications into both execution paths:**

In `run_yolo_version()`, added `start_time = time.monotonic()` before the results
loop, then after `_print_version_report()`:
```python
notifier = Notifier(cfg)
if notifier.enabled() and not dry_run:
    payload = _build_notify_payload(version, results, "completed", start_time, cfg)
    notifier.send(payload)
```

In `main()`, after `run_implement()` returns:
```python
notifier = Notifier(config)
if notifier.enabled() and not dry_run:
    notifier.send({
        "project": proj["name"],
        "spec": spec.stem if spec else "interactive",
        "status": "completed",
    })
```

**Bug 2 — require both Telegram env vars:**
```python
# after
def enabled(self) -> bool:
    telegram_ready = bool(self._tg_token_env and self._tg_chat_env)
    return bool(self._webhook_url or telegram_ready or self._script)
```

`send()` was also tightened to gate `_send_telegram()` on both vars:
```python
if self._tg_token_env and self._tg_chat_env:
    self._send_telegram(payload)
```

## Tests added

- `test_notifier_enabled_false_with_only_telegram_token` — asserts `enabled()` is
  False when only `telegram_token_env` is set
- `test_run_yolo_version_calls_notifier_on_completion` — patches `Notifier.send` and
  verifies it is called with the correct payload after a successful version run
- `test_main_calls_notifier_after_run_implement` — patches `Notifier.send` and
  verifies it is called after `run_implement()` returns

## Rules for future work

1. When adding a new class or function, trace every caller path from `main()` to
   confirm the code is actually reachable — "defined but never called" is a silent
   no-op, not a compile error.
2. For multi-part config (e.g., Telegram needs both token AND chat_id), `enabled()`
   must require ALL parts, not just one. A channel that can never fire is not
   "enabled".
3. Never check partial Telegram config in isolation — always test both the enabled
   flag and the send path together in the same test.
