# fix039 — dry_run guard missing from atexit handler and completion notification

**Date:** 2026-05-16

## Symptom

Running `learnx_dk.py --dry-run` still sent a real webhook/Telegram notification on
completion (or an "aborted" notification on Ctrl-C), even though no Docker session was
actually started.

## Root cause

When notifications were wired into `run_yolo_version()` (see fix038), the dry_run guard
was applied correctly to the main completion send but missed two other execution paths:

**Path 1 — atexit handler:**
```python
# before
def _atexit_handler() -> None:
    if _notified[0] or not notifier.enabled():
        return
    payload = _build_notify_payload(...)
    notifier.send(payload)
```
The atexit handler fires on any exit, including dry-run exits.

**Path 2 — completion send in run_yolo_version():**
```python
# before
if notifier.enabled():
    payload = _build_notify_payload(...)
    notifier.send(payload)
    _notified[0] = True
```
The dry_run flag was in scope but not checked here.

Both paths called `notifier.send()` regardless of whether a real run had occurred.

## Fix

Added `or dry_run` to both guard conditions:

```python
# atexit handler
def _atexit_handler() -> None:
    if _notified[0] or dry_run or not notifier.enabled():
        return
    ...

# completion send
if notifier.enabled() and not dry_run:
    payload = _build_notify_payload(...)
    notifier.send(payload)
    _notified[0] = True
```

## Tests added

- `test_completion_notification_skipped_in_dry_run` — verifies `Notifier.send` is not
  called when `run_yolo_version()` is invoked with `--dry-run`
- `test_atexit_handler_skipped_in_dry_run` — verifies the atexit handler exits early
  when dry_run is True

## Rules for future work

1. When a flag like `dry_run` must suppress side effects, search for every code path
   that produces those side effects — not just the obvious one. Functions registered
   via `atexit` are easy to miss.
2. Treat all three notification sites as a unit: atexit, completion, and error paths
   must all carry the same guard, or add an assertion in the `Notifier` constructor that
   it rejects calls when dry_run is set globally.
