# fix031 — --version flag silently bypassed the selected mode

**Date:** 2026-05-16

## Symptom

Calling `main(["--mode", "supervised", "--version", "v5"])` would execute the full
yolo path (Docker container + E2E tests + review pipeline) while `_parse()` still
returned `mode="supervised"` in the 5-tuple. The mode value in the returned tuple
was inaccurate — it did not match the execution path actually taken.

Similarly, `--version v5` with no `--mode` flag would default-parse to
`mode="supervised"` and then silently use yolo execution anyway.

## Root cause

`main()` dispatched `--version` before the mode `if/elif` block:

```python
if version:
    run_yolo_version(...)   # always yolo, ignores mode entirely
    return

if not dry_run:
    _print_banner(mode)     # mode value here was never updated
```

The parsed mode was never updated to reflect what actually ran, so the 5-tuple
returned by `_parse()` was self-inconsistent when `--version` was combined with
any mode other than `"yolo"`.

## Fix

Added a normalization step at the end of `_parse()`, after all flag parsing and
mutual-exclusion checks:

```python
if version is not None:
    mode = "yolo"
```

This ensures the returned 5-tuple always has `mode="yolo"` when `--version` is
present, regardless of what `--mode` was passed. Execution and parsing now agree.

## Tests added

- `test_version_normalizes_mode_to_yolo` — verifies `--mode supervised --version v5`
  returns `mode="yolo"`.
- `test_version_without_mode_normalizes_to_yolo` — verifies `--version v5` alone
  (default mode path) also returns `mode="yolo"`.

## Rules for future work

1. When adding a new flag that implies a specific mode, normalize `mode` inside
   `_parse()` — not inside `main()`. The 5-tuple should be self-consistent so
   callers never need to second-guess mode vs. dispatch logic.
2. Any flag that forces a specific execution path should be tested for the case
   where an incompatible `--mode` is explicitly provided alongside it.
