# fix028 — OpenRouter client had no HTTP timeout: requests hung silently

**Date:** 2026-05-15

## Symptom

When using `--provider openrouter`, a generation run appeared to hang indefinitely
on a specific LLM call with no error and no timeout. With no progress output (see
fix027) the hang was indistinguishable from normal slow operation.

## Root cause

The OpenAI Python SDK default timeout is 600 seconds (10 minutes). The OpenRouter
client in `tutor/infra/llm.py` was constructed without a `timeout` parameter:

```python
return OpenAI(
    api_key=config.openrouter_api_key,
    base_url="https://openrouter.ai/api/v1",
    default_headers={"HTTP-Referer": "http://localhost"},
    # no timeout — SDK default is 600s
)
```

A slow or unresponsive model could hold the connection open for up to 10 minutes
before the SDK gave up.

## Fix

Added `timeout=120.0` to the OpenRouter client construction:

```python
return OpenAI(
    api_key=config.openrouter_api_key,
    base_url="https://openrouter.ai/api/v1",
    default_headers={"HTTP-Referer": "http://localhost"},
    timeout=120.0,
)
```

120 seconds is generous for any single LLM response. The retry logic (`retry_count`)
then handles transient failures.

## Rules for future work

1. Always set `timeout` when constructing an OpenAI-compatible client for a third-party
   provider — the SDK default of 600s is too long to wait silently.
2. The Groq client does not have this problem in practice (responses are fast), but
   consider adding a timeout there too if Groq ever shows slowdowns.
3. `retry_count` and `retry_delay_s` in `llm_config.toml` control retry behaviour
   after a timeout — keep `retry_count` at 3 for OpenRouter.
