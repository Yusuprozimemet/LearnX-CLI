# fix041 — Subprocess exit codes swallowed: failed review session exits 0

**Date:** 2026-05-16

## Symptom

When the Claude subprocess inside `run_phase1()` or `run_phase2()` exited with a
non-zero code (e.g. Docker not available, Claude rate-limited, permission denied),
`run_review.py` still exited 0. Shell scripts and CI treating non-zero as failure would
not detect the problem.

## Root cause

Two discard points in the original `main()`:

```python
# before
_rc1, phase1_output, had_findings = run_phase1(...)
# _rc1 stored with underscore prefix but never checked

if two_phase and had_findings:
    run_phase2(...)           # return value not captured at all
elif ...
```

`_rc1` with a leading underscore is a Python convention for "intentionally unused". The
reviewer assumed the return code was irrelevant, but a failing Claude session (rc=1)
looks identical to a successful one from the shell's perspective.

`run_phase2()` was called in statement form — its `(rc, output)` tuple was completely
dropped.

## Fix

Extracted phase orchestration into `_run_phases()`, which propagates both codes:

```python
def _run_phases(...) -> int:
    rc1, phase1_output, had_findings = run_phase1(...)
    if rc1 != 0:
        return rc1                   # surface phase 1 failure immediately

    if two_phase and had_findings:
        rc2, _ = run_phase2(...)
        return rc2                   # surface phase 2 failure
    ...
    return 0

def main(...) -> None:
    ...
    rc = _run_phases(...)
    if rc != 0:
        sys.exit(rc)
```

The refactor also brought `main()` under the 40-line limit by separating CLI parsing
(`_parse_argv`) and config resolution (`_resolve_config`) into their own functions.

## Tests added

No new tests for exit-code propagation were added (the subprocess is always mocked in
unit tests). The behaviour is verified manually: a mock `run_phase1` returning `(2, "", False)`
causes `main()` to exit with code 2.

## Rules for future work

1. Never store a subprocess return code with a leading underscore (`_rc`). If you don't
   intend to act on it, document why. If the code could be non-zero, propagate it.
2. Treat a function call whose return value is a `(returncode, ...)` tuple as a
   statement only when you have a deliberate reason to ignore the code — and leave a
   comment saying so.
3. When extracting orchestration logic into a helper, declare its return type as
   `-> int` and have `main()` call `sys.exit()` on non-zero. This makes the contract
   explicit and auditable.
