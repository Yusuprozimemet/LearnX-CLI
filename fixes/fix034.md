# fix034 — run_yolo_version() ignored config image/workspace for every spec run

**Date:** 2026-05-16

## Symptom

When `devloop.toml` contained a custom `docker_image` or `workspace`, running
`--version v5` ignored those values. Every spec in the version loop launched with
the hardcoded `IMAGE = "learnx-dev"` and `WORKSPACE = "/workspace"` constants,
not the values read from config.

## Root cause

`run_yolo_version()` received `specs_dir` from `main()` (added in day22) but was
never given `image` or `workspace` parameters. Its call to `run_implement()` at
line 323 omitted both:

```python
run_implement(
    project_dir, home_dir, spec=spec, review=review,
    extra_args=extra_args, dry_run=dry_run
    # ← image and workspace missing; run_implement used its own defaults
)
```

`main()` correctly extracted `image` and `workspace` from config and passed them
to the single-spec `run_implement()` path, but the `--version` path went through
`run_yolo_version()` which never received them.

## Fix

Added `image: str = IMAGE` and `workspace: str = WORKSPACE` parameters to
`run_yolo_version()` and forwarded them in the `run_implement()` call:

```python
def run_yolo_version(..., image: str = IMAGE, workspace: str = WORKSPACE) -> None:
    ...
    run_implement(
        project_dir, home_dir, spec=spec, review=review,
        extra_args=extra_args, dry_run=dry_run,
        image=image, workspace=workspace,   # ← forwarded
    )
```

Updated `main()` to pass `image` and `workspace` to `run_yolo_version()`.

## Test added

`test_run_yolo_version_forwards_image_to_run_implement` — patches `run_implement`
and `_checkout_spec_branch`, calls `run_yolo_version()` with `image="custom-img"`
and `workspace="/custom"`, then asserts those values appear in the mock's kwargs.

## Rules for future work

1. When adding a new config-driven parameter to any function, audit every call site
   in the call chain — not just the first caller. Config values must flow all the
   way to the function that actually uses them.
2. Any function that delegates to another function that accepts config params must
   accept and forward those same params, even if it does not use them directly.
