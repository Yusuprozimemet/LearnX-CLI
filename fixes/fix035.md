# fix035 — _build_e2e_command() hardcoded IMAGE/WORKSPACE, causing mixed environments

**Date:** 2026-05-16

## Symptom

When `run_implement()` was called with a custom `image` or `workspace` (e.g. from
`devloop.toml`), the main container session and the E2E smoke tests ran in different
environments:

- Container session: `custom-image` mounted at `/custom`
- E2E tests: `learnx-dev` mounted at `/workspace` (hardcoded constants)

This could cause E2E tests to pass or fail against a stale or wrong image, and
would produce misleading results — the tests would not be validating the same
container that ran the implementation.

## Root cause

`_build_e2e_command()` was never parameterised. It referenced the module-level
constants `IMAGE` and `WORKSPACE` directly:

```python
def _build_e2e_command(project_dir: pathlib.Path) -> list[str]:
    return [
        "docker", "run", "--rm",
        "-v", f"{_to_posix(project_dir)}:{WORKSPACE}",   # ← hardcoded
        "-w", WORKSPACE,                                  # ← hardcoded
        IMAGE,                                            # ← hardcoded
        ...
    ]
```

When `run_implement()` was updated to accept `image`/`workspace` in day22, the
E2E call inside it was not updated alongside.

## Fix

Added `image: str = IMAGE` and `workspace: str = WORKSPACE` to `_build_e2e_command()`
and replaced the three constant references with the parameters:

```python
def _build_e2e_command(
    project_dir: pathlib.Path,
    image: str = IMAGE,
    workspace: str = WORKSPACE,
) -> list[str]:
    return [
        "docker", "run", "--rm",
        "-v", f"{_to_posix(project_dir)}:{workspace}",
        "-w", workspace,
        image,
        ...
    ]
```

Both call sites in `run_implement()` (dry-run path and live path) were updated to
pass `image=image, workspace=workspace`.

## Test added

`test_implement_review_dry_run_e2e_uses_custom_image` — calls `run_implement()` in
dry-run mode with `image="custom-img"` and `workspace="/custom"`, then asserts both
values appear in the printed output (which includes the E2E command).

## Rules for future work

1. Any helper that builds a Docker command must accept `image` and `workspace` as
   parameters — never read module-level constants directly inside a docker-command
   builder.
2. When a function is updated to accept config-driven params, search for all internal
   helpers it calls that also build Docker commands and update them in the same change.
