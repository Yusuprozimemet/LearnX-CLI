# fix025 — yolo / container mode requires a real TTY; cannot launch from inside Claude Code

**Date:** 2026-05-15

## Symptom

Running the yolo launcher from inside a Claude Code session (via the Bash tool)
produced one of these errors:

```
the input device is not a TTY
```
or
```
cannot attach stdin to a TTY-enabled container because stdin is not a terminal
```

The container exited immediately without showing the Claude Code prompt.

## Root cause

The `docker run` command uses `-it` (interactive + TTY) so that Claude Code inside
the container gets a proper terminal:

```python
cmd = ["docker", "run", "--rm", "-it", ...]
```

The `-t` flag allocates a pseudo-TTY. When the parent process (the Bash tool inside
Claude Code) does not itself have a TTY attached — as is the case for any
programmatically launched subprocess — Docker cannot allocate one and aborts.

This is a Docker/OS constraint, not a bug in the launcher.

## Fix

**No code change.** The launcher must be run from a real terminal.

Run yolo from a PowerShell window opened directly (Win+X → Terminal), not from
inside a Claude Code session or any other subprocess context.

```powershell
# Open a new PowerShell window, then:
.\.venv\Scripts\Activate.ps1
python scripts/learnx_dk.py --mode yolo --spec specs/v4/dayN.md
```

The `--dry-run` flag does not spawn Docker and can safely be run from anywhere
(including from inside Claude Code via the Bash tool) to preview the commands.

## Rules for future work

1. Always launch `container` and `yolo` modes from a real PowerShell terminal,
   never from inside a Claude Code session or a CI pipeline.
2. Use `--dry-run` to verify the command looks correct before launching.
3. If a CI pipeline ever needs to run the container mode non-interactively,
   remove `-t` (keep `-i`) from `build_docker_command`. Do not do this for
   interactive use — it breaks the terminal experience inside the container.
